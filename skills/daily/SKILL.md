# Daily

## 前置约束

写入任何 raw/wiki 文件前，**必须先读取** `/mnt/c/obsidian_wiki/CLAUDE.md` 获取契约和路径规范。

若存在训练计划文件：

- `/home/DataTransfer/Pyrojewel/vscode/coros-workout-mcp/output/training-schedule.json`

则在生成 daily 时**优先读取当天训练安排**，把训练强度和预计时长纳入日程判断。

## 触发
用户输入 `/daily`

## 执行流程

1. **确定日期**：`date_str = YYYY-MM-DD`（当前日期）

2. **读取昨日 daily 记录**（如果存在）：
   - 路径：`/mnt/c/obsidian_wiki/daily/{yesterday}.md`
   - 优先读取 `## Todo`、`## Done`、`## Review`
   - 提取未完成 TODO 作为"继承建议"列表

3. **读取当日训练计划**（如果存在）：
   - 路径：`/home/DataTransfer/Pyrojewel/vscode/coros-workout-mcp/output/training-schedule.json`
   - 识别 `date_str` 当天是否有训练
   - 将训练分为三类：
     - `轻唤醒`：如 `5k 晨跑`，可优先放早上
     - `中等训练`：如 `8k/10k 轻松跑 + 力量`、`10k + 短加速`，通常更适合傍晚或固定训练时段
     - `长距离`：如 `15k`，优先放早上
   - 若用户已明确给出固定时间偏好（例如“周六 14:00-16:00 和队友训练”），优先遵守用户偏好，不自行改动

4. **检查今日文件状态**：
   - 路径：`/mnt/c/obsidian_wiki/daily/{date_str}.md`
   - **不存在** → 新建文件（跳到步骤 5）
   - **存在** → 读取现有 `Todo / Done / Review`，询问是否修改

5. **若修改现有计划**：
   - 展示现有内容
   - 询问要修改/添加/删除哪些项
   - 更新文件

6. **若新建文件**：
   - 询问"今天的主线任务 / 唯一重点是什么？"
   - 展示昨日继承建议（如果有）："昨日未完成的 TODO：..."
   - 若存在训练计划，展示当日训练摘要
   - 基于主线任务 + 训练安排 + 节律规则，帮助用户压成当天 TODO
   - 写入文件

## 生成 daily 的默认思路

daily 不是简单抄用户一句 TODO，而是要把当天压成**可执行的一天**。默认按下面顺序组织：

1. **主线优先**
   - 当天必须有一个最重要的主线任务
   - 避免把多个大目标平铺进同一天

2. **上午优先**
   - 能放早上的关键事项尽量放早上
   - 上午默认优先给：
     - 晨跑/长距离（如果适合）
     - 第一段深度工作
     - 主线最难任务

3. **训练感知**
   - `5k 唤醒跑`：通常适合早上
   - `8k/10k + 力量`：通常放固定训练时段，不再额外加晨跑
   - `10k + 短加速`：当天晚上不再安排高强度脑力攻坚
   - `15k 长距离`：优先放早上；长跑后当天只做轻整理和复盘

4. **节律锚点**
   - daily 里的 TODO 默认尽量覆盖这几个锚点：
     - 训练 / 身体唤醒
     - 一段深度工作
     - 一段执行或整理
     - 晚间收尾

5. **不把一天写满**
   - daily 应保留余量
   - 不要把“理想中的完美一天”写进去

## 默认约束

若用户没有额外说明，daily 默认带入这些约束：

- `23:00` 上床
- 上午优先做主线
- 训练后的晚上只做轻整理，不做硬攻坚
- 消息/杂事/人际尽量放下午处理
- 不把手机、短视频、擦边内容写成休息方式
- 若当天状态差，也至少保住：
  - 主线一小步
  - 训练或轻运动
  - 按时收住

## 文件模板

```markdown
# {date_str}

## Todo
- [ ] {今日主线任务}
- [ ] {训练或身体锚点}
- [ ] {一段深度工作 / 执行任务}
- [ ] {收尾或复盘}

## Done
- {完成项}

## Review
- {简短回顾}
```

说明：

- `Todo` 仍然只保留一个扁平列表，不额外增加 section
- 但条目内容应体现当天主线、训练安排和节律锚点
- 若训练时间明确，可直接写进 TODO，例如：
  - `14:00-16:00 和队友完成 10k 轻松跑 + 核心力量`
  - `早上完成 15k 长距离，下午只做周复盘`

## 边界

- `daily` 只保留当天 `todo / done / review`
- 不写入 `wiki/synthesis/daily/`
- 不承载项目碎片、截图、参数变化、调试细节
- 这些内容先走 `inbox/notes/projects/{project}/` 或项目专属 `inbox/{project}/logs/`，后续再由 `inbox-prepare` / `wiki-capture` 提升到 `raw/notes/projects/{project}/logs/YYYY-MM-DD-{project}-log.md`

## 规则
- 所有写入使用 Read + Write/Edit 工具，不执行脚本
- 写入前必须读取 CLAUDE.md 确认规范
- 不要把 `daily` 当作直接入库源；`daily` 属于根目录个人运行层
- 不要把项目细节、实验参数、调试碎片塞进 daily
- daily 应该帮助用户把一天“收拢”，而不是继续摊大饼

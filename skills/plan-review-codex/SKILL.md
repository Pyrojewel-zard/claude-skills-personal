# Plan Review Codex: 多轮方案审阅循环

对已有的 plan 文档进行多轮审阅，确保没有过度设计和核心纰漏。

## 输入

- **Plan 路径**：由 `$ARGUMENTS` 传入，可以是：
  - Superpowers plan 文件（如 `~/.claude/plans/xxx.md`）
  - 项目 docs 目录下的 plan 文件
  - 任意 markdown 格式的方案文档

## 常量

- **MAX_ROUNDS = 3** — 最大审阅轮数
- **REVIEWER_MODEL = `gpt-5.4`** — Codex MCP 使用的模型
- **OUTPUT_DIR = `plan-review-out/`** — 审阅输出目录（与 plan 文件同级）

## 审阅维度

### 过度设计检测

- 不必要的抽象层
- 过早优化
- 冗余的中间步骤
- 过于复杂的依赖关系
- 可简化的流程

### 核心纰漏检测

- 遗漏的关键步骤
- 缺失的依赖声明
- 逻辑漏洞或矛盾
- 未考虑的边界情况
- 缺失的验证/测试步骤

## 工作流程

### 初始化

1. **解析输入路径**：
   - 从 `$ARGUMENTS` 提取 plan 文件路径
   - 若路径不存在或无法解析 → 报错退出

2. **读取 plan 内容**：
   - 读取完整的 plan 文档
   - 提取 frontmatter（如有）和正文结构

3. **创建输出目录**：
   - 在 plan 文件同级目录创建 `plan-review-out/`
   - 初始化 `REVIEW_LOG.md`

### 循环（最多 3 轮）

#### Phase A: 调用 Codex 审阅

使用 `mcp__codex__codex` 工具调用 Codex：

```json
{
  "model": "gpt-5.4",
  "prompt": "你是一位资深的软件架构师和方案评审专家。请审阅以下方案文档。\n\n## 审阅要求\n\n1. **过度设计检测**：识别不必要的复杂性、过早优化、冗余步骤\n2. **核心纰漏检测**：识别遗漏的关键步骤、缺失依赖、逻辑漏洞\n\n## 方案文档\n\n[完整 plan 内容]\n\n## 输出格式\n\n请按以下格式输出：\n\n### 审阅结论\n- **状态**：通过 / 需修改\n- **置信度**：高 / 中 / 低\n\n### 发现的问题\n\n#### 过度设计（如有）\n1. [问题描述] - [严重程度：高/中/低]\n   - 建议简化方式：...\n\n#### 核心纰漏（如有）\n1. [问题描述] - [严重程度：高/中/低]\n   - 建议修复方式：...\n\n### 修改建议\n\n[具体的修改建议，可直接应用到文档]\n\n---\n\n**重要**：如果方案没有过度设计且无核心纰漏，请明确输出「状态：通过」。\n如有任何需要修改的问题，请输出「状态：需修改」并提供具体修改建议。"
}
```

#### Phase B: 解析审阅结果

从 Codex 响应中提取：

- **状态**：`通过` 或 `需修改`
- **问题列表**：分类为过度设计和核心纰漏
- **修改建议**：具体的修改内容

**终止条件**：
- 状态为「通过」→ 结束循环，输出最终结果
- 达到 MAX_ROUNDS → 结束循环，输出当前状态和剩余问题

#### Phase C: 自动修复

若状态为「需修改」：

1. **解析修改建议**：
   - 提取具体的修改操作（增/删/改）
   - 确定修改位置（章节/段落）

2. **应用修改**：
   - 使用 Edit 工具修改 plan 文件
   - 保留原有 frontmatter 和结构
   - 在修改处添加注释标记：`<!-- reviewed by codex, round N -->`

3. **记录修改**：
   - 追加到 `REVIEW_LOG.md`

#### Phase D: 记录本轮

追加到 `plan-review-out/REVIEW_LOG.md`：

```markdown
## Round N (timestamp)

### Codex 审阅结果

<details>
<summary>完整响应</summary>

[Codex 原始响应]

</details>

### 问题摘要

| 类型 | 问题 | 严重程度 | 状态 |
|------|------|----------|------|
| 过度设计 | ... | 高/中/低 | 已修复/待处理 |
| 核心纰漏 | ... | 高/中/低 | 已修复/待处理 |

### 已应用的修改

- [修改 1]
- [修改 2]

### 状态

- [继续下一轮 / 审阅通过 / 达到最大轮数]
```

### 终止

当循环结束时：

1. **更新 REVIEW_LOG.md**：
   - 添加最终状态摘要
   - 若未通过，列出剩余问题

2. **输出结果**：
   - **通过**：告知用户方案已通过审阅
   - **未通过**：展示剩余问题和建议

## 输出示例

### 审阅通过

```
✅ 方案审阅通过

经过 2 轮审阅，Codex 确认方案：
- 无过度设计
- 无核心纰漏

修改记录已保存至：plan-review-out/REVIEW_LOG.md
```

### 达到最大轮数

```
⚠️ 审阅未完全通过

经过 3 轮审阅，仍有以下问题：

### 剩余问题

1. [问题 1] - 建议：...
2. [问题 2] - 建议：...

### 建议

- 手动处理剩余问题
- 或调整方案后重新运行审阅

修改记录已保存至：plan-review-out/REVIEW_LOG.md
```

## 关键规则

- **必须使用 `gpt-5.4` 模型**（遵循 codex-mcp.md 规则）
- 每轮审阅前读取最新的 plan 内容（已修复的版本）
- 修改时保留原有格式和结构
- 不删除用户手动添加的内容
- 审阅日志必须完整记录每轮的原始响应

## 错误处理

- **Codex 调用失败**：重试 1 次，若仍失败则记录错误并退出
- **修改冲突**：记录冲突位置，跳过该修改，继续下一轮
- **无法解析响应**：记录原始响应，请求用户手动处理
